## Контекст: почему рефакторим

Code Agent сейчас — склейка трёх React-шеллов с двумя несвязанными понятиями «проект»
и историей чата в localStorage. Это даёт три подтверждённых дефекта и общее ощущение
«сырого, непонятного» инструмента. Цель — **chat-first**: чат — главная поверхность и
единственный источник контекста; проект/файлы/git/RAG — панели, которые **следуют** за
активной сессией, а не живут своей жизнью.

### Карта текущей архитектуры (as-is)

```
App.tsx
└─ EliraChatShell (mainTab: "chat" | "code")
   ├─ "chat": обычный чат + левый сайдбар (Чаты / Проекты / Файлы / …)
   │           "Проекты" → ProjectPanel  (backend "advanced project" сессия)
   │           "Файлы"   → library (RAG-документы для чата)
   └─ "code": CodeWorkspaceShell
              ├─ Left:   список сессий (свой)
              ├─ Center: CodeAgentChatShell   ← история в localStorage (!)
              └─ Right:  drawer → IdeWorkspaceShell(forceView)  ← "advanced project" (!)
```

### Подтверждённые дефекты (с привязкой к коду)

1. **Переключение чатов ломается** — `CodeAgentChatShell.tsx`.
   Два эффекта дерутся на смену `sessionId`:
   - стр. 494-497: `saveHistoryFor(sessionId, history)` deps `[history, sessionId]`
   - стр. 501-510: `setHistory(loadHistoryFor(sessionId))` deps `[sessionId]`
   При клике на другой чат `sessionId` уже новый, а `history` ещё старый → эффект
   сохранения пишет **старую** историю под ключ **нового** чата, эффект загрузки
   читает её обратно. Новый чат показывает чужие сообщения и не чистится; localStorage
   нового чата портится навсегда.

2. **«Проект открыт слева, не открыт справа»** — два независимых понятия проекта:
   - `projectRoot` (строка-путь, тулбар Code Agent) → инструменты агента + RAG-индекс.
   - «advanced project» (backend-сессия, `openAdvancedProject`) → дерево/Git правой панели.
   `IdeWorkspaceShell` в drawer'е рендерится **без** проп `projectRoot`
   (`CodeWorkspaceShell.tsx:1436-1444`), а filetree грузится через
   `getAdvancedProjectTree` (`IdeWorkspaceShell.tsx:467-469`), который требует
   отдельного `openAdvancedProject`. CodeWorkspaceShell его никогда не вызывает →
   правая панель всегда «Проект не открыт» (`IdeWorkspaceShell.tsx:695-698`).

3. **Auto-index «Failed to fetch»** — `CodeWorkspaceShell.tsx:563-586`.
   Эффект бьёт `api.startProjectWatcher` при монтировании (если тумблер был сохранён
   включённым). Если backend ещё не поднялся — сетевая ошибка + пугающий тост. Нет
   ретрая/проверки готовности. Отдельно: **`watchdog` отсутствует в requirements**
   (импортится в `backend/app/application/code_agent/file_watcher.py:35-36`, на чистой
   установке auto-index упадёт).

### Важный факт для WS1

Бэкенд **уже умеет хранить историю**: таблица `sessions` имеет колонку `turns_json`,
а `update_session` принимает ключ `turns` (`backend/app/application/code_agent/sessions.py:38-189`).
Фронт пишет turns на бэкенд **только один раз** при миграции legacy-истории
(`CodeWorkspaceShell.tsx:272-273`), а в рабочем режиме гоняет историю в localStorage.
То есть механизм персистентности готов — надо просто им пользоваться.

## WS1 — История чата на бэкенде (убивает дефект №1 в корне) — ✅ DONE `7d04403`

**Цель:** backend `turns` — единственный источник правды. localStorage-история удаляется
как механизм (остаётся только одноразовая миграция legacy-ключей).

**Файл:** `frontend/src/components/CodeAgentChatShell.tsx`

**Сделать:**
1. При смене `sessionId` грузить историю из бэкенда: `const full = await api.getCodeSession(sessionId)` →
   `setHistory(full.turns ?? [])`. Это асинхронно — добавить guard (ref с «текущим
   запрошенным sessionId»), чтобы поздний ответ от прежней сессии не перезаписал свежую.
2. Персистить историю на бэкенд **после завершения хода** (не на каждый токен стрима):
   когда стрим/тул-ран завершился и `history` финализирован — `api.patchCodeSession(sessionId, { turns })`.
   Дебаунс не нужен, но дублирующие патчи подряд недопустимы — патчить один раз на ход.
3. **Удалить** racey-эффект сохранения `[history, sessionId]` (стр. 494-497) и функции
   `saveHistoryFor` / `loadHistoryFor` как механизм рабочей персистентности.
4. Эффект на `[sessionId]` (стр. 501-510) оставить для abort/reset live-стейта
   (`abortRef.abort()`, `setRunning(false)`, `setInput("")`), но историю он теперь
   берёт из бэкенда (п.1), а не из localStorage.
5. Скролл вниз при изменении истории вынести в отдельный эффект `[history]` (без `sessionId`),
   чтобы не тащить за собой логику сохранения.

**Файл:** `frontend/src/components/CodeWorkspaceShell.tsx`
- Legacy-миграция (стр. 271-305) остаётся: если в localStorage есть старые ключи
  `elira_code_agent_history_v2_*`, перенести их в backend `turns` и **удалить ключи** после
  успешного патча. После переноса localStorage больше не используется для истории.

**НЕ делать:** не вводить новую таблицу/эндпоинт — `getCodeSession`/`patchCodeSession`/`turns`
уже всё покрывают.

**Критерий готовности:** переключение между чатами и «Новый чат» показывает корректную
историю каждого; перезагрузка страницы сохраняет историю; в localStorage не остаётся
ключей истории после миграции; гонки нет (проверить быстрым переключением туда-обратно).
